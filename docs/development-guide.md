# 개발 가이드

상태: guide

## 목적

이 문서는 HSA AI 파트를 팀원이 같은 방식으로 개발하기 위한 기본 규칙을 정리한다.

## 초기 개발 원칙

- 전체 연결 방향은 `Frontend <-> Backend <-> AI`로 둔다.
- AI는 백엔드하고만 통신한다.
- AI는 고객 응답을 직접 전송하지 않고, 답변 초안과 최소 판단 정보를 백엔드에 반환한다.
- 문의 저장, DB 조회, 상태 관리는 백엔드 책임으로 둔다.
- 정책 문서와 답변 생성 지식은 AI 파트 책임으로 둔다.
- 코드보다 API 계약과 책임 경계를 먼저 합의한다.
- 실제 LLM, DB, VectorDB 연결은 adapter 뒤에 둔다.
- 초기에는 mock 또는 stub으로 흐름을 검증한다.
- 애매한 문의는 관리자 검토가 필요하다는 결과로 반환한다.
- secret은 git에 커밋하지 않는다.

## Git Flow

### main

- 프로젝트 최종 merge 브랜치
- 기본 프로젝트 세팅 및 배포 가능한 브랜치
- 항상 배포 가능한 상태를 유지한다.

### develop

- 기능 개발 결과를 통합하는 브랜치
- 개발 브랜치는 작업 완료 후 develop 브랜치로 PR을 남긴다.
- main으로 merge하기 전 통합 검증을 수행한다.

### {type}/{issue number}

- 개발 브랜치
- 예: `feat/#5`, `fix/#11`, `docs/#12`, `chore/#3`
- 작업 단위로 이슈 생성 후 브랜치를 생성한다.
- 생성한 브랜치에서 작업을 완료한 뒤 develop 브랜치로 PR을 남긴다.
- 모든 작업 시작 전, 생성한 브랜치에서 develop 브랜치를 pull 받은 후 작업한다.

## PR Rules

- Assignee에는 본인을 지정한다.
- Reviewers에는 본인을 제외한 팀원 전원을 지정한다.
- PR 생성 후 카카오톡으로 공유한다.
- 팀원 1명 이상이 코멘트를 달면 PR 작성자가 직접 승인 후 merge한다.
- merge된 브랜치는 자동 삭제되며, 필요 시 복구 가능하다.

PR 설명에는 다음 내용을 포함한다.

- 작업 목적
- 변경한 파일
- 구현 범위
- 제외한 범위
- 확인한 명령 또는 검증 결과
- 리뷰어가 특히 봐야 할 부분

## Commit Message Convention

커밋 메시지는 아래 형식을 따른다.

```text
Type: commit title
```

예시:

```text
Feat: 로그인 기능 추가
Docs: AI 파트 API 계약 초안 추가
Chore: Python 프로젝트 초기 설정 추가
```

| Type | Description |
| --- | --- |
| Feat | 새로운 기능 추가 |
| Fix | 버그 수정 |
| Docs | 문서 수정 |
| Style | 코드 formatting, 세미콜론 누락 등 코드 자체의 변경이 없는 경우 |
| Refactor | 코드 리팩토링 |
| Test | 테스트 코드, 리팩토링 테스트 코드 추가 |
| Chore | 패키지 매니저 수정, 그 외 기타 수정 예: `.gitignore` |
| Comment | 필요한 주석 추가 및 변경 |
| Rename | 파일 또는 폴더명을 수정하거나 옮기는 작업만인 경우 |
| Remove | 파일을 삭제하는 작업만 수행한 경우 |
| Init | 프로젝트 초기 세팅 |
| Merge | 브랜치 merge |
| !BREAKING CHANGE | 커다란 API 변경의 경우 |
| !HOTFIX | 급하게 치명적인 버그를 고쳐야 하는 경우 |

## PR 전 체크리스트

- secret이 포함되지 않았다.
- `.env` 파일이 커밋되지 않았다.
- README 또는 docs가 변경 내용과 맞다.
- API 계약이 바뀌었다면 `api-contract-v2.md`를 수정했다.
- 새로운 의존성을 추가했다면 이유를 설명했다.
- 구현 코드가 있다면 실행 또는 테스트 결과를 남겼다.
